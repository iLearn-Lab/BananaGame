# 配角生成问题解决方案清单

> **基于**：阿衍、拍短片的等配角图生成的异常案例分析  
> **目标**：系统性修复「主角误判为配角」「主图含多人」「body_ref 异常」等问题

---

## 实施记录（已落地）

| 日期 | 改动 |
|------|------|
| 2025-02 | P0/P1/P2 方案已实现：建档过滤主角、身份揭示排除主角、剧情 prompt 强化、vision 整图跳过 body_ref、appearance_hints 强化、初登场图 prompt 强化 |

---

## 一、问题与根因速览

| 序号 | 现象 | 根因 |
|------|------|------|
| 1 | 阿衍 body_ref/主图异常 | 主图生成失败或抽象；vision 在异常/多人图上裁错人；外观描述不足 |
| 2 | 拍短片的初登场图包含主角 | 被误建档为配角，初登场场景为主角+配角同框，整图被当参考 |
| 3 | 主角称呼「拍短片的」被当成配角 | 建档流程无主角姓名/别名校验，完全信任剧情模型输出 |
| 4 | 角色名与别名冲突 | 身份揭示可能误把主角称呼并入配角；建档未排除主角 |

## 二、解决方案清单

### 2.1 建档前过滤主角称呼（对应问题 3、4）

#### 目标
在 `plot_supporting_characters` 进入建档流程前，排除主角本人及其别名，避免将主角误建档为配角。

#### 方案 A：建档入口校验（推荐）

**位置**：`game_server.py` → `_archive_supporting_roles_on_option_shown`

**逻辑**：
1. 从 `global_state` 获取 `protagonist_canonical`（主角规范信息）
2. 抽取主角姓名（`name_zh`）、英文名（`name_en`），以及世界观中可能的主角别称
3. 遍历 `plot_supporting_characters` 时，若 `display_name` 与主角姓名/别名匹配，则跳过建档并打日志

**实现要点**：
```python
# 伪代码
protagonist_names = set()
if canonical := global_state.get("protagonist_canonical"):
    for k in ["name_zh", "name_en"]:
        if v := canonical.get(k):
            protagonist_names.add(str(v).strip())
# 可从 core_worldview.characters["主角"] 中再取常见别称

for display_name, slot in plot_supporting_characters:
    if display_name in protagonist_names:
        print(f"⏭️ 跳过建档：{display_name} 为主角称呼，不建配角档案")
        continue
    # 原有建档逻辑...
```

**依赖**：建档时需能访问 `global_state` 中的 `protagonist_canonical`；若此处只有 `option_data`，需在调用链中传入或在 `option_data` 中附带主角信息。

---

#### 方案 B：剧情解析后置过滤

**位置**：`src/story/options.py`，在解析出 `plot_supporting_characters` 后

**逻辑**：
1. 解析完成后，调用「主角名称/别名集合」获取函数
2. 过滤掉与主角匹配的 `(name, slot)`，再写入 `option_data` 和 `global_state`

**优点**：统一在数据源处过滤，下游建档、生图都不会再收到主角称呼  
**缺点**：需在 options 流程中拿到主角信息；预生成、补图等路径需保持一致

---

#### 方案 C：强化剧情模型 prompt

**位置**：`src/story/options.py`、`src/llm/local_gen.py` 等剧情生成 prompt

**逻辑**：
1. 在【本段出场配角】的说明中，明确写入主角姓名（如「主角名为沈屿白，请勿将沈屿白或拍短片的等主角称呼列入配角」）
2. 在 system_msg 或 user prompt 中增加：「若角色是玩家视角/主角本人，绝不能写入【本段出场配角】」

**优点**：从源头减少模型犯错  
**缺点**：无法完全杜绝，仍需方案 A/B 作为兜底

---

**建议**：方案 A + 方案 C 组合使用。

---

### 2.2 身份揭示时排除主角（对应问题 4）

#### 目标
`update_supporting_role_aliases_from_plot` 在更新配角别名时，不应把主角的称呼加入配角 aliases；也不应把主角相关揭示写进配角档案。

#### 方案

**位置**：`src/characters/supporting.py` → `update_supporting_role_aliases_from_plot`

**逻辑**：
1. 调用前获取主角姓名/别名集合（可从 game 的 core_worldview 或 global_state 读取）
2. 对每条「原名|新身份」：
   - 若 `orig` 或 `new_id` 属于主角称呼，跳过该条
   - 若将导致「主角称呼 → 配角 alias」或「配角 alias → 主角称呼」，同样跳过

**实现要点**：
```python
# 需传入 protagonist_names 或从 game_id 加载世界观获取
if orig in protagonist_names or new_id in protagonist_names:
    continue  # 不更新
```

**依赖**：该函数需能访问主角信息，可能需增加参数 `protagonist_names: Optional[Set[str]] = None`，由调用方传入。

---

### 2.3 初登场图尽量单人/聚焦配角（对应问题 2）

#### 目标
降低「配角初登场图含主角」的概率，使 body_ref 裁剪有更清晰的单人画面。

#### 方案 A：图片提示词强化

**位置**：`src/image/prompt_optimize.py`、`api_providers.py` 中生成场景图提示词的位置

**逻辑**：
1. 当本段有「首次出场」配角时，在提示词中增加约束：
   - 「该配角首次登场，尽量让其在画面中占据主要位置、或单独构图」
   - 「若为双人/多人场景，明确写出配角在图中的位置特征（如左侧、持相机者）」
2. 与现有「明确配角对应图中哪个人物」的要求衔接，不冲突

**优点**：不改流程，只改 prompt  
**缺点**：生图模型未必严格 obey，效果有限

---

#### 方案 B：建档时检测多人场景（预警）

**位置**：`src/characters/supporting.py` → `archive_supporting_role_first_appearance`

**逻辑**：
1. 调用 vision 模型时，可额外返回「图中检测到的人数」
2. 若人数 > 1，记录日志或标记 `_multi_person_first_img: true`，便于后续人工复查或自动重裁

**实现成本**：需扩展 vision 接口，或增加一次轻量检测调用

---

#### 方案 C：vision 裁剪失败时降级策略

**位置**：`src/characters/supporting.py` → `archive_supporting_role_first_appearance`  
**相关**：`vision_ref_crop.py` → `_call_vision_bbox`

**逻辑**：
1. 当 vision 返回 `0 0 1 1`（整图）或 bbox 异常时，视为「无法可靠裁出单人」
2. 不写入 `face_ref_path`，仅使用整张初登场图作参考，并在档案中标记 `_body_ref_fallback: true`
3. 后续生图流程若发现 `_body_ref_fallback`，可优先依赖外观描述而非可能错误的裁剪图

---

**建议**：优先实施方案 A；方案 B、C 可视资源情况作为增强。

---

### 2.4 body_ref 与主图生成质量（对应问题 1）

#### 目标
提升配角主图可用性，以及 vision 裁剪时能正确裁出该配角本人。

#### 方案 A：强化 vision 的 appearance_hints

**位置**：`src/characters/supporting.py` → `archive_supporting_role_first_appearance`  
**相关**：`vision_ref_crop.py` → `_call_vision_bbox`

**逻辑**：
1. 将 `appearance_hints` 扩展为：`first_prompt`（含该角色的核心描述）+ `core_features` + 可选 `first_appear_scene` 片段
2. 对 `core_features` 做关键词抽取（如体型、服饰、姿态），使 prompt 更聚焦该角色
3. 在 vision prompt 中明确写出：「排除主角（若已知主角外貌），只框选名为 XXX 的配角」

**实现要点**：
- 若建档时已知主角外貌简述，可写入 vision prompt：`图中可能有主角（XXX），请只框选配角「阿衍」`

---

#### 方案 B：主图异常检测与重试

**位置**：生图流程返回后、入库/复制前

**逻辑**：
1. 可选：对生成的主图做简单异常检测（如亮度分布、边缘检测、或调用轻量 vision 做「是否含清晰人体」判断）
2. 若判定为抽象/糊片/无效图，触发重试或标记 `_img_quality_warning`
3. 若 body_ref 从该类图裁出，可标记不信任，降级为用整图或仅用文本描述

**实现成本**：需定义「异常」标准，并接入检测逻辑

---

#### 方案 C：vision 返回 0 0 1 1 时跳过 body_ref

**位置**：`src/characters/supporting.py`、`vision_ref_crop.py`

**逻辑**：
1. `_call_vision_bbox` 返回 `{"x":0,"y":0,"width":1,"height":1}` 时，视为「找不到人/不确定」
2. 不进行裁剪，或裁剪后不写入 `face_ref_path`，避免把整张废图/多人图当 body_ref

---

#### 方案 D：角色名与外貌绑定加强

**位置**：剧情模型输出、prompt_optimize 的配角参考说明

**逻辑**：
1. 【本段出场配角】格式可扩展为：`角色名-配角N|简短外貌关键词`（可选）
2. 建档时将该外貌片段写入 `core_features`，供 vision 和生图使用
3. 减轻「同名多角色」或「外貌描述模糊」导致的裁剪/生图错误

---

**建议**：方案 A + 方案 C 优先；方案 B、D 作为后续优化。

---

### 2.5 建档时机与数据可用性

#### 目标
确保建档时能获取主角信息，用于过滤和 vision 增强。

#### 方案

**位置**：`game_server.py`、调用 `_archive_supporting_roles_on_option_shown` 的链路

**逻辑**：
1. 在 `notify-scene-displayed` 等触发建档的请求中，确保 `global_state` 或 `option_data` 携带：
   - `protagonist_canonical`
   - 或 `core_worldview.characters["主角"]` 的摘要
2. 若当前请求体无这些字段，根据 `game_id` 从缓存/数据库加载世界观，再执行建档

## 五、验收标准

1. **主角过滤**：剧情输出「拍短片的-配角2」且拍短片的=主角时，不建立配角档案，并打日志
2. **身份揭示**：不将主角称呼加入任何配角的 aliases
3. **body_ref 质量**：vision 返回整图 bbox 时不写入 face_ref_path，避免废图当参考
4. **主图与裁剪**：首次出场配角的初登场图与 body_ref，在多数场景下能正确对应该配角本人
