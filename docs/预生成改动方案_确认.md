# 预生成改动方案确认

## 目标

1. **首屏也预生成**：第一次选项生成完成后，服务端立即启动首屏选项的预生成（与后续轮一致），不依赖前端稍后触发的 `pregenerate-next-layers`。
2. **流水线化**：
   - 上一轮**剧情文本**生成完 → **立即**开始下一轮该分支的剧情文本 + 生图提示词生成（不等待本层图片）。
   - **下一层文本**生成完 → **立即**开始该分支下一层的图片预生成（图片是对应剧情文本生成的，必须先有文本再生图）。

---

## 一、首屏预生成

### 现状

- `generate_initial_options()` 只生成：初始场景文本 + 一张场景图 + `initial_options`，**不**预生成每个选项的剧情/图片。
- 首屏预生成依赖前端在「文本开始显示」后调用 `pregenerate-next-layers`，且前端自己生成 `sceneId`，存在延迟与不一致风险。

### 改动

| 位置 | 改动内容 |
|------|----------|
| **game_server.py** `generate_initial_options()` | 在写入 `initial_cache` 并 `events['main'].set()` 之后：<br>1. 生成 `server_scene_id = generate_scene_id(str(global_state), str(initial_options))`<br>2. 将 `initial_cache['pregeneration_scene_id'] = server_scene_id`<br>3. 调用 `_pregenerate_next_layers_logic(global_state, initial_options, server_scene_id)`，在后台启动首屏预生成（与后续轮同一套逻辑） |
| **game_server.py** `generate_option()` 返回 initial 数据时 | 当 `scene_id in (None, 'initial')` 且从 `initial_cache` 返回 `option_data` 时，在返回的 JSON 中增加：<br>`"sceneId": initial_cache.get("pregeneration_scene_id")`<br>（若存在则前端用此 ID 作为当前场景 ID，与预生成缓存一致） |
| **game-frontend** 首次拿到选项数据时 | 若接口返回里带有 `sceneId`（即 `optionData.sceneId`），则：<br>`gameState.currentSceneId = optionData.sceneId`<br>调用 `pregenerateNextLayers` 时优先使用该 `sceneId`（即首屏用服务端下发的 ID，不再仅用前端生成的 `generateNewSceneId()`） |

### 效果

- 首屏在服务端「第一次选项」完成时即开始预生成，无需等前端展示后再发请求。
- 前端用服务端下发的 `sceneId` 请求 `generate-option`，能命中同一 `scene_id` 的预生成缓存。
- 前端仍可照常调用 `pregenerate-next-layers`（传同一 `sceneId`），后端可做幂等（已有该 scene 的预生成则直接返回或跳过）。

## 二、流水线：上一轮文本完 → 立即下一轮文本；上一层图完 → 立即下一层图

### 现状

- 第一层：多选项并行（如 2 worker），每个选项内部「先生成文本 → 再生成图片」串行；**全部**第一层任务完成后才启动**整段**第二层逻辑。
- 第二层：`generate_layer2()` 等待「第一层所有选项的文本都写入缓存」后，再为每个分支依次调用 `generate_all_options(..., skip_images=False)`（文本+图一起）。

### 目标行为

- **某个选项 i 的剧情文本一写完** → 立刻为该分支启动「下一轮」：只生成下一轮剧情文本 + 生图提示词（即下一层 layer2 的文本），写入 `next_scene_id` 的 layer1（text_only），**不**等本层选项 i 的图片完成。
- **某个分支的下一层文本一写完** → 立刻为该分支启动「下一层图片」预生成：对 `next_scene_id` 下刚写入的、缺图的那几条（text_only）按剧情文本生成图片并回写缓存。**图片必须对应剧情文本生成，因此必须先有 layer2 文本再生成 layer2 图片。**

这样：

- 下一层文本可以和本层图片并行，不必等本层所有图都出完。
- 下一层文本一完成，对应下一层图片马上开始，不依赖「本层图是否完成」；生图顺序正确（先文本后图）。

### 改动（核心逻辑在 server/pregeneration.py，若 game_server 内仍有重复实现则同步）

| 项目 | 改动要点 |
|------|----------|
| **1. 第一层单选项任务内：文本写入缓存后** | 在 `generate_single_option_task` 中，在「文本写入 `cache_entry['layer1'][opt_idx]` 且设为 `text_completed`、并 `events[opt_idx].set()`」之后，**立即**启动「仅该分支」的第二层**文本**预生成（不阻塞当前任务，新起线程）：<br>• 读取 `layer1_option_data = cache_entry['layer1'][opt_idx]`、`next_options`、用 `flow_update` 得到 `updated_global_state`，计算 `next_scene_id`<br>• 调用 `generate_all_options(updated_global_state, next_options, skip_images=True)`<br>• 将结果写入 `pregeneration_cache[next_scene_id]['layer1']`，状态为 `text_only`，并记录 `cache_entry['layer2_next_scene_id'][opt_idx] = next_scene_id`（供后续「本层图完 → 下一层图」使用）<br>• 触发对应 `generation_events`（若有其他逻辑依赖） |
| **2. 第二层文本写缓存后（同一分支）** | 在「仅该分支」的 layer2 **文本**预生成完成（`generate_all_options(..., skip_images=True)` 已执行完毕，结果已写入 `pregeneration_cache[next_scene_id]['layer1']`，状态为 `text_only`）之后，**立即**在同一逻辑内或紧接着启动「仅该分支」的第二层**图片**预生成（新起线程或同线程内顺序执行）：<br>• 对刚写入的 `pregeneration_cache[next_scene_id]['layer1']` 中状态为 `text_only`、且无 `scene_image.url` 的条目，按剧情文本逐个调用 `generate_scene_image`，写回 `scene_image` 并将状态改为 `completed`<br>• 图片是对应剧情文本生成的，因此必须在 layer2 文本完成后再生图<br>• 注意限速、锁与 `cache_key_suffix`（如 `f"{next_scene_id}_opt{next_opt_idx}"`）与现有逻辑一致 |
| **3. 取消「等整层第一层再跑整段 layer2」** | 删除或不再依赖「等所有第一层任务 `future.result()` 完成后再启动一个统一的 `generate_layer2()` 线程」的逻辑。第二层改为：**文本**由「第一层某选项文本完 → 该分支 layer2 文本」触发；**图片**由「该分支 layer2 文本写完 → 该分支 layer2 图片」触发。若存在「用户已选择某选项则只生成该选项的第二层」的语义，保留并在 spawn 出的线程内根据 `layer2_selected_option` 判断。 |
| **4. 兼容与边界** | • 若某分支 `next_options` 为空，不启动该分支的 layer2 文本/图。<br>• layer2 图片一定在「该分支 layer2 文本写完后」再启动，保证图片对应剧情文本生成。<br>• 保留 `layer2_cancel`、`layer2_selected_option` 等现有取消/单选逻辑，在 spawn 出的线程内检查。 |

### 数据流简述

- **第一层选项 i 文本完成**  
  → 写 `layer1[i]`、`text_completed`  
  → 起线程：`generate_layer2_text_for_branch(i)` → `generate_all_options(..., skip_images=True)` → 写 `pregeneration_cache[next_scene_id]`（text_only）  
  → **紧接着**（同一分支）起线程或继续：`generate_layer2_images_for_branch(i)` → 对 `next_scene_id` 下刚写入的 text_only 条目按剧情文本逐个生图并更新为 `completed`。  
  → 即：layer2 文本一写完就触发 layer2 图片，不依赖「本层选项 i 的图是否完成」；图片始终对应剧情文本生成。

## 三、涉及文件与顺序建议

| 顺序 | 文件 | 说明 |
|------|------|------|
| 1 | **game_server.py** | 首屏：`generate_initial_options` 里写 `pregeneration_scene_id` 并调用 `_pregenerate_next_layers_logic`；`generate_option` 返回 initial 时带 `sceneId` |
| 2 | **game-frontend (script-modular.js)** | 首屏：若 `optionData.sceneId` 存在则设 `gameState.currentSceneId`，预生成调用时优先用该 ID |
| 3 | **server/pregeneration.py** | 流水线：在 `generate_single_option_task` 中文本写缓存后 spawn「仅该分支的 layer2 文本」；layer2 文本写完后紧接着 spawn「仅该分支的 layer2 图片」（先文本后图）；移除或弱化「等整层再跑 generate_layer2」 |
| 4 | **game_server.py**（若仍包含重复的 `_pregenerate_next_layers_logic`） | 与 server/pregeneration.py 保持一致：要么删除重复实现并统一用 `from server.pregeneration import _pregenerate_next_layers_logic`，要么把上述流水线改动同步一份到 game_server 内的实现 |

   - 不再等「整层第一层完成」再跑整段第二层。
