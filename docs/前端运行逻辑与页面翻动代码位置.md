# 前端运行逻辑与页面翻动代码位置

本文档用于说明当前 Web 前端的运行流程、页面切换逻辑，以及“页面翻动/翻页”相关代码的具体位置，便于后续排查与修改。

## 1. 前端入口与整体结构

当前 Web 前端核心逻辑集中在以下 3 个主文件（同时配有多个动效/配色样式文件）：

- `game-frontend/index.html`
- `game-frontend/script-modular.js`
- `game-frontend/style.css`

其中职责划分如下：

- `index.html`：定义所有页面容器和主要交互节点，例如主菜单、属性选择页、难度页、基调页、主题输入页、图片风格页、设定页、加载页、剧情页、存档页、结局页。
- `script-modular.js`：前端主逻辑文件，负责初始化、状态管理、界面切换、调用后端接口、剧情分段展示、选项生成、存档读写等。
- `style.css`：负责基础布局、核心动画、选项卡样式、打字机效果、右下角箭头按钮、隐藏/显示区域样式等。
- `transitions.css`：负责舞台化屏幕转场（`screen-entering` / `screen-leaving`）和屏幕级入场动画。
- `color_system.css`：负责全局设计 token、语义化组件类和动态主题（`mood-*`）背景。

页面初始化入口在：

- `game-frontend/script-modular.js` 中的 `document.addEventListener('DOMContentLoaded', ...)`
- 入口调用 `Game.init()`

对应逻辑位置：

- `script-modular.js`：`Game.init()` 定义在开头
- `script-modular.js`：`DOMContentLoaded` 初始化在文件末尾

## 2. 当前前端运行主链路

当前前端运行顺序如下：

1. 浏览器加载 `index.html`
2. `script-modular.js` 在 `DOMContentLoaded` 后执行 `Game.init()`
3. `Game.init()` 依次执行：
   - `initSoundManager()`
   - `initGameState()`
   - `initElements()`
   - `initEventListeners()`
4. 初始页面默认为主菜单 `menu-screen`
5. 用户从主菜单开始，按顺序进入：
   - 属性选择页
   - 难度选择页
   - 基调选择页
   - 主题输入页
   - 图片风格选择页
   - 加载页
   - 设定展示页
   - 再进入剧情页 `gameplay-screen`
6. 进入剧情页后，前端通过后端接口拉取首段剧情与选项，并进入“剧情文本分段展示 -> 用户点箭头继续 -> 显示选项 -> 用户选项 -> 请求下一剧情”的循环

## 3. 状态管理的核心位置

前端所有流程都围绕 `gameState` 运转。

核心初始化位置：

- `game-frontend/script-modular.js` 中的 `initGameState()`

`gameState` 当前保存的关键数据包括：

- `currentScreen`：当前显示的页面
- `selectedDifficulty`：已选难度
- `selectedTone` / `currentTone`：已选故事基调 / 当前结局基调
- `protagonistAttr`：主角属性
- `gameTheme`：主题输入
- `imageStyle`：图片风格
- `gameData`：后端返回的全局状态，包含 `core_worldview` 与 `flow_worldline`
- `currentScene`：当前剧情文本
- `currentOptions`：当前剧情选项
- `currentSceneId`：当前场景 ID，用于后端缓存和预生成命中
- `lastSceneImage`：上一段剧情图片，用于后续生成连续视觉内容
- `textSegments`：当前剧情拆分后的文本段
- `currentTextSegmentIndex`：当前显示到哪一段
- `isShowingSegments`：当前是否处于“分段翻页”状态
- `pendingOptions`：待展示的选项，等最后一段文字看完再显示

这部分是理解“页面翻动”逻辑的基础，因为剧情页并不是一次性把完整文字和选项都展示出来，而是先拆段、逐段推进。

## 4. 页面切换逻辑

### 4.1 页面切换的实现方式

当前项目的“页面切换”不是路由跳转，而是单页应用内多个 `screen` 容器切换显示。

页面容器定义位置：

- `game-frontend/index.html`

主要页面节点包括：

- `menu-screen`
- `attr-selection-screen`
- `difficulty-selection-screen`
- `tone-selection-screen`
- `theme-input-screen`
- `image-style-selection-screen`
- `setting-screen`
- `loading-screen`
- `gameplay-screen`
- `save-management-screen`
- `ending-screen`

页面切换核心函数位置：

- `game-frontend/script-modular.js` 中的 `switchScreen(screenName)`

### 4.2 `switchScreen()` 的逻辑

`switchScreen(screenName)` 的执行流程如下：

1. 遍历 `elements.screens`，移除 `active`，并添加 `hidden` + `screen-leaving`
2. 找到目标页面后移除 `hidden`，再添加 `active` + `screen-entering`
3. 把当前页面名写回 `gameState.currentScreen`
4. 根据页面节点的 `data-screen-theme` 更新 `body` 上的 `theme-*` 类
5. 根据目标页面名更新 `body` 上的 `mood-*` 类，用于背景/粒子动态联动
6. 对部分页面做额外处理：
   - 进入 `themeInput` 时清空主题输入框
   - 进入 `imageStyleSelection` 时重置风格选择状态
   - 进入 `saveManagement` 时自动调用 `loadSaves()`
   - 进入 `gameplay` 时显示角色面板

样式配合位置：

- `game-frontend/style.css` 中 `.screen`
- `game-frontend/transitions.css` 中 `.screen.screen-entering` / `.screen.screen-leaving`

因此，前端第一层“翻页”本质上是 `switchScreen()` 驱动的多屏幕切换。

## 5. 菜单到剧情页的运行链路

### 5.1 菜单与配置阶段

主要事件绑定位置：

- `game-frontend/script-modular.js` 中的 `initEventListeners()`

关键按钮流转如下：

- 点击 `start-btn` -> `switchScreen('attrSelection')`
- 属性确认 -> `switchScreen('difficultySelection')`
- 难度确认 -> `switchScreen('toneSelection')`
- 基调确认 -> `switchScreen('themeInput')`
- 主题提交 -> `switchScreen('imageStyleSelection')`
- 风格确认 -> `switchScreen('loading')`，随后调用 `generateGameWorldview()`

### 5.2 世界观生成阶段

世界观生成函数位置：

- `game-frontend/script-modular.js` 中的 `generateGameWorldview()`

它会向后端请求：

- `POST http://127.0.0.1:5001/generate-worldview`

发送内容包括：

- `gameTheme`
- `protagonistAttr`
- `difficulty`
- `toneKey`
- `imageStyle`

成功后会把后端返回的 `globalState` 写入：

- `gameState.gameData`

同时把设定页内容填充到这些 DOM 节点中：

- `game-style-content`
- `worldview-content`
- `protagonist-ability-content`
- `chapter-conflict-content`

### 5.3 进入首段剧情

用户在设定页点击“开始游戏”后：

- 调用 `simulateGameLoading()`
- 再进入 `continueToFirstScene()`

`continueToFirstScene()` 里会：

1. `switchScreen('gameplay')`
2. 从 `gameState.gameData.flow_worldline` 取当前章节
3. 更新顶部章节标题和核心矛盾
4. 调用后端接口生成首段剧情：
   - `POST http://127.0.0.1:5001/generate-option`
   - 初始参数中的 `option` 固定为 `开始游戏`
   - `sceneId` 为 `null`
5. 拿到剧情和选项后，交给 `displayScene()` 展示

因此，剧情页真正的运行入口不是 HTML 自动渲染，而是：

- `continueToFirstScene()` -> 后端返回首段内容 -> `displayScene()`

## 6. 当前“页面翻动”逻辑是什么

当前项目里，“页面翻动”实际上分成两层：

- 第一层：配置页、加载页、设定页、剧情页之间的整页切换
- 第二层：剧情页内部，文本按句切段后，用户点击右下角箭头逐段推进

通常如果你说的“页面翻动”是“剧情中一段一段往后翻”，那核心逻辑在第二层。

## 7. 剧情页内部翻动逻辑

### 7.1 剧情文本先被拆段

文本拆段函数位置：

- `game-frontend/script-modular.js` 中的 `splitTextIntoSegments(text)`

它会按句末标点进行切分，主要识别：

- `。`
- `！`
- `？`
- 以及标点后跟闭引号的情况

处理结果会写入：

- `gameState.textSegments`
- `gameState.currentTextSegmentIndex`
- `gameState.isShowingSegments`

### 7.2 `displayScene()` 是剧情页展示核心

核心函数位置：

- `game-frontend/script-modular.js` 中的 `displayScene(text, options, imageData, videoData, optionDataForArchive)`

这个函数负责：

1. 保存当前剧情文本和选项到 `gameState`
2. 调用 `splitTextIntoSegments(text)` 把文本拆段
3. 记录 `pendingOptions`，让选项先不要立刻展示
4. 若有图片则更新全局背景图；若没有图则异步调用补图接口
5. 显示文本区域，隐藏选项区域
6. 只先展示第一段文字
7. 用打字机效果逐字渲染
8. 当前段展示完后，显示右下角箭头按钮，并通过 `setNextSegmentPromptState()` 标记“继续下一段/展示选项”
9. 调用 `animateSegmentStage()` 给文本区/选项区注入分镜入场动画

关联的 DOM 位置在：

- `game-frontend/index.html`
- 文本区域：`text-display-area`
- 文本元素：`scene-text`
- 选项区域：`options-list-area`
- 箭头按钮：`next-segment-btn`

关联样式位置在：

- `game-frontend/style.css` 中 `.typewriter`
- `game-frontend/style.css` 中 `.text-display-area.hidden`
- `game-frontend/style.css` 中 `.options-list-area.hidden`
- `game-frontend/style.css` 中 `.scene-arrow`
- `game-frontend/style.css` 中 `.scene-arrow.hidden`

### 7.3 右下角箭头如何决定“继续下一段”还是“显示选项”

这是当前“翻页”最关键的实现点。

控制逻辑依赖：

- `next-segment-btn.dataset.showOptions`

工作方式如下：

1. `displayScene()` 显示第一段文字
2. 如果后面还有下一段：
   - 显示箭头
   - 设置 `dataset.showOptions = 'false'`
3. 如果已经是最后一段：
   - 显示箭头
   - 设置 `dataset.showOptions = 'true'`

这意味着：

- `false`：点击箭头后继续翻到下一段文字
- `true`：点击箭头后不再显示文字，而是切到选项区

相关代码位置：

- `displayScene()` 中段落结束后的判断逻辑 + `setNextSegmentPromptState()`
- `showNextTextSegment()` 中段落结束后的同类判断逻辑 + `setNextSegmentPromptState()`
- `initEventListeners()` 里给 `next-segment-btn` 绑定的点击事件

### 7.4 用户点击箭头后的行为

箭头按钮事件位置：

- `game-frontend/script-modular.js` 中 `initEventListeners()` 末尾附近

执行逻辑：

1. 点击 `next-segment-btn`
2. 读取 `nextSegmentBtn.dataset.showOptions`
3. 如果为 `true`：
   - 隐藏箭头
   - 隐藏文本区
   - 显示选项区，并触发 `animateSegmentStage({ toOptions: true })`
   - 从 `gameState.pendingOptions` 读取待展示选项
   - 调用 `generateOptions()`
   - 清空本轮分段状态
4. 如果为 `false`：
   - 调用 `showNextTextSegment()`

因此，当前剧情翻页的核心调度关系是：

- `displayScene()`：启动第一页
- `showNextTextSegment()`：翻到后续页
- `next-segment-btn click`：决定“继续翻页”还是“转到选项”

### 7.5 `showNextTextSegment()` 的作用

函数位置：

- `game-frontend/script-modular.js` 中的 `showNextTextSegment()`

它负责：

1. `currentTextSegmentIndex + 1`
2. 清空当前文本节点内容
3. 对下一段继续执行打字机效果
4. 判断是否还有更多段
5. 若还有更多段，箭头继续保持“下一段”模式
6. 若已经到最后一段，箭头切换为“显示选项”模式

可以理解为：

- `displayScene()` 负责“打开当前剧情第一页”
- `showNextTextSegment()` 负责“在同一剧情内部继续翻页”

## 8. 用户选项后的下一轮运行逻辑

选项生成函数位置：

- `game-frontend/script-modular.js` 中的 `generateOptions(options)`

它的流程是：

1. 隐藏文本区
2. 显示选项区
3. 根据 `options` 动态生成 `.option-card`
4. 用户点击某个选项后，显示加载遮罩
5. 向后端发起下一轮剧情请求

调用接口：

- `POST http://127.0.0.1:5001/generate-option`

关键请求参数包括：

- `option`
- `globalState`
- `optionIndex`
- `sceneId`
- `previousSceneId`
- `previousSceneImage`
- `previousSceneText`

后端返回后，前端会：

1. 读取 `optionData.scene`
2. 读取 `optionData.next_options`
3. 读取 `optionData.scene_image`
4. 更新 `gameState.currentSceneId`
5. 更新 `gameState.gameData.flow_worldline`
6. 更新章节进度条
7. 再次调用 `displayScene(cleanedNextScene, nextOptions, validatedSceneImage, null, optionData)`

所以完整循环是：

- 选项点击 -> `/generate-option` -> `displayScene()` -> 分段翻页 -> 箭头 -> 选项 -> 再次请求

## 11. 结论：当前“页面翻动”真正由谁控制

如果只看剧情页内部的“翻页/翻动”，当前真正控制逻辑的核心是以下 4 个点：

1. `splitTextIntoSegments(text)`：把剧情拆成多段
2. `displayScene(...)`：显示第一段，并决定箭头初始状态
3. `showNextTextSegment()`：点击后继续翻到下一段
4. `next-segment-btn.dataset.showOptions`：决定箭头点击后是“继续翻页”还是“展示选项”

换句话说，当前项目的“页面翻动”并不是使用现成分页组件或轮播组件实现，而是：

- 前端自行切段
- 通过 `gameState` 记录段落索引
- 通过右下角按钮手动推进
- 在最后一段结束后切换到选项列表

如果后续要修改翻页体验，优先查看以下代码：

- `game-frontend/script-modular.js` 中的 `displayScene(...)`
- `game-frontend/script-modular.js` 中的 `showNextTextSegment()`
- `game-frontend/script-modular.js` 中 `next-segment-btn` 的点击事件
- `game-frontend/index.html` 中剧情页的文本区/选项区/箭头按钮结构
- `game-frontend/style.css` 中 `.typewriter` 与 `.scene-arrow` 相关样式
