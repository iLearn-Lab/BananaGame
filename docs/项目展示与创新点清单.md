# 项目展示与创新点清单

本文档**完整展示**本项目目前做了什么、核心亮点以及创新点所在，便于对外介绍、复盘或评审参考。

---

## 一、项目定位与一句话概括

**本项目是一款 AI 驱动的沉浸式文本冒险游戏**：用户输入主题与偏好，由大模型生成完整世界观、章节矛盾、剧情与选项，配合多供应商图像生成（主角立绘、场景图），支持**命令行**与 **Web 双端**运行；具备**结构化世界观、流动世界线、配角动态建档、信息差剧情、结局预测与基调修改**等完整叙事与角色体系。

---

## 二、目前做了什么（功能清单）

### 2.1 运行与入口

| 能力 | 说明 |
|------|------|
| **双运行模式** | **CLI**：`main.py` / `main2.py` 在终端内文字冒险；**Web**：`game_server.py` + `game-frontend/` 在浏览器中玩。 |
| **统一逻辑层** | 核心生成逻辑集中在 `src/` 与 `main2.py`，CLI 与 Web 共用同一套世界观生成、选项生成、结局、图片逻辑。 |
| **跨平台启动** | `启动游戏.bat`（Windows）、`启动游戏.sh`（Linux/macOS）一键启动；Windows 下 UTF-8 防乱码已处理。 |

### 2.2 世界观与剧情生成

| 能力 | 说明 |
|------|------|
| **全局世界观生成** | 根据用户主题、主角属性、难度、基调，由 LLM 生成 **core_worldview**（风格、主线、章节矛盾、主角/配角设定等）与 **flow_worldline**（当前章节、矛盾状态等）。 |
| **分阶段与缓存** | 支持世界观模板、缓存（按主题/属性/难度/基调 key）、后台异步补全完整世界观，兼顾速度与完整性。 |
| **单步剧情生成** | 根据当前 global_state、用户选择的选项序号或文本指令，生成下一段**场景描述** + **2 个新选项**，并输出 **flow_update**（含章节矛盾是否解决）、**深层背景关联**、**本段出场/提及配角**。 |
| **结构化输出与解析** | 剧情输出约束为【场景】【选项】【世界线更新】【深层背景关联】【本段出场配角】【本段提及但未出场】；代码内正则 + 格式约束 + 代码块剥离，失败有兜底与调试信息。 |
| **难度与基调** | 内置**简单/中等/困难**难度；多种**故事基调**（如 HE/BE/NE、黑暗宿命、幽默、唯美、逻辑推理、克苏鲁式神秘等），每种有详细叙事逻辑、文风、角色与禁忌规则，用于约束 LLM 生成风格。 |

### 2.3 章节与结局系统

| 能力 | 说明 |
|------|------|
| **章节矛盾** | **flow_worldline** 中维护 `current_chapter`、`chapter_conflict_solved`；LLM 在【世界线更新】中可标记「章节矛盾：已解决/未解决」；每完成一章触发**章节深化**（角色深层背景深化）。 |
| **结局预测** | 开局或关键节点调用 `generate_ending_prediction()`，生成**隐藏的结局主基调 + 结局大致内容**，供后续基调修改与最终结局生成使用。 |
| **结局基调修改** | 在**深层背景节点**触发时（如重要人物死亡、核心羁绊断裂等）调用 `modify_ending_tone()`，根据触发事件调整 HE/BE/NE 等主基调；`modify_ending_content()` 根据当前进度小幅调整结局内容。 |

### 2.4 角色系统

| 能力 | 说明 |
|------|------|
| **主角属性与形象** | 用户选择主角属性（如性别、年龄、性格等）；生成世界观后**依赖世界观**生成主角立绘（正面/侧/背三视图），保证与设定一致；LLM 优化主角形象提示词后再调用图像 API。 |
| **配角动态识别与建档** | 从剧情中 **extract_supporting_characters_in_scene** / **extract_supporting_characters_with_names** 识别配角；**展示后触发建档**（避免凭空生成）；**get_or_create_supporting_role_archive**、**archive_supporting_role_first_appearance**、**update_supporting_role_aliases_from_plot**（从剧情中提取身份揭示，如「黑衣人就是艾玛」）维护配角档案。 |
| **配角参考图** | 配角首次出场或建档后可生成参考图；支持视觉裁单人图、名解析加固；预生成第二层时也可生成配角相关图。 |
| **深层背景** | 配角与主角均可具备**深层背景**（隐藏设定）；选项可**关联角色深层背景**，选择后**解锁**该深层背景；解锁后后续剧情需**围绕已解锁深层背景展开**，自然融入主线而非直接抛设定。 |

### 2.5 信息差与剧情深化

| 能力 | 说明 |
|------|------|
| **信息差记录** | **flow_worldline** 中维护 **info_gap_record**（已记录的信息差条目）；选择「关联深层背景」的选项会记录信息差并可能触发结局基调修改。 |
| **信息差阈值剧情** | 当**未发现的信息差数量达到阈值（如 5 条）**时，自动调用 LLM 生成**隐藏的剧情深化内容**（揭秘、反转、前后呼应），自然嵌入常规剧情。 |
| **章节深化** | 每完成一个章节，对当前涉及角色进行**深层背景深化**（多级深度），使角色设定更丰富、更贴合主线；深化后同样记录信息差并可触发结局基调修改。 |

### 2.6 选项与预生成（Web 体验）

| 能力 | 说明 |
|------|------|
| **每步 2 选项** | 每步生成 2 个选项，经 **prune_options** 过滤不合理或过相似选项。 |
| **预生成两层** | Web 端支持 **pregenerate-next-layers**：在用户看到当前层选项时，后台预生成**第一层 + 第二层**选项与剧情（可含场景图），存入 **pregeneration_cache**；用户选择第一层后通过 **get-pregenerated-layer2** 直接取第二层数据，**显著减少等待时间**。 |
| **预生成图片** | 预生成时可一并生成并缓存场景图，避免用户选完选项后再长时间等图。 |
| **文本兜底** | 图失败或关闭时仍可玩**纯文本模式**，不阻塞主线。 |

### 2.7 图像生成

| 能力 | 说明 |
|------|------|
| **多供应商** | 支持 **云雾（yunwu）**、**Replicate**、**OpenAI DALL·E**、**Stable Diffusion**（本地/远程）、**ComfyUI**、**图生图（img2img）** 等，通过 **IMAGE_GENERATION_CONFIG** 与 .env 配置切换。 |
| **当前顶模** | 主剧情 LLM：**Camera_Analyst_MODEL**（如 Claude Opus 4.6）；图像生成：**Image_Generation_MODEL**（如 Gemini 3 Pro Image Preview），均支持通过 .env 配置。 |
| **提示词优化** | 场景图、主角立绘均先经 **LLM 优化提示词**（optimize_image_prompt_with_llm、optimize_main_character_prompt_with_llm），保证风格一致、与剧情/主角设定一致。 |
| **主角三视图** | 主角正面图 + 侧/背视图（可走文生图或图生图）；配角参考图、场景图按需生成；支持按视口或配置计算生成尺寸。 |
| **图片缓存** | 场景图可按 prompt_hash 缓存到本地，减少重复请求。 |

### 2.8 百科与现实题材增强

| 能力 | 说明 |
|------|------|
| **Wikipedia 检索** | 对用户输入主题做 **wiki_lookup_theme_and_character**：调用 Wikipedia 中英文 API（搜索、摘要、跨语言标题、图片 URL）；二次关键词检索可能的主角/人物；若判定为现实题材/IP，将 **Wikipedia 主题条目** 与 **二次检索** 结果作为证据注入 LLM/图像提示词，**增强现实与 IP 题材的准确性与代入感**。 |

### 2.9 存档与前端

| 能力 | 说明 |
|------|------|
| **存档** | **save-game**、**list-saves**、**load-game**、**delete-save**，CLI 与 Web 共用 **saves/** 目录。 |
| **前端** | 单页结构（主菜单 → 属性 → 难度 → 基调 → 主题 → 世界观生成 → 设定展示 → 图片风格 → 玩法）；剧情文本 + 选项列表 + 下一段；角色状态面板、存档管理、结局界面；Tailwind、Font Awesome、音效与字体管理、无障碍与兼容逻辑。 |

## 三、核心亮点

1. **全流程 AI 叙事闭环**  
   从「用户主题 + 属性 + 难度 + 基调」到「世界观 → 章节矛盾 → 每步剧情与选项 → 配角识别与深层背景 → 信息差剧情 → 结局预测与基调修改」，全部由 LLM + 图像模型驱动，且用**结构化 prompt 与解析**保证可维护与可扩展。

2. **双端一致体验**  
   CLI 与 Web 共用同一套 `src/` 与 main2 逻辑，存档互通；Web 通过**预生成两层选项 + 预生成图片**显著降低等待感，同时保留**文本兜底**保证可玩性。

3. **叙事机制完整**  
   **flow_worldline**（章节、矛盾状态、信息差记录）与 **core_worldview** 分离，既有一致的世界观，又有随选择流动的状态；**深层背景关联选项** + **信息差阈值剧情** + **章节深化** + **结局预测/基调修改** 形成「隐藏信息 → 揭秘/反转 → 影响结局」的完整链条。

4. **角色与设定可追溯**  
   主角立绘**依赖世界观生成**；配角**动态识别、建档、身份揭示**（含从剧情中抽取别名）；深层背景**分级解锁、自然融入后续剧情**，避免一次性信息轰炸。

5. **基调与难度可配置**  
   多种故事基调（HE/BE/NE、黑暗宿命、幽默、唯美、逻辑推理、克苏鲁式神秘等）具备细粒度叙事逻辑、文风与禁忌，直接约束 LLM 输出风格；难度影响剧情容错率与矛盾解决难度。

6. **多供应商与顶模就绪**  
   LLM 与图像均可通过 .env 配置；当前设计为**顶模**（如 Claude Opus 4.6 + Gemini 3 Pro Image Preview），并配有**图像模型切换清单**与 **AI 使用说明**，便于合规与成本统计。

7. **现实题材增强**  
   Wikipedia 主题与人物检索可注入世界观与图像提示词，对现实/IP 题材的细节与一致性有明确增强。

## 四、创新点

1. **「流动世界线」+ 章节矛盾 + 结局预测联动**  
   将「当前章节、矛盾是否解决、信息差记录」显式纳入状态（flow_worldline），并在每一步剧情与选项中由 LLM 输出 **flow_update**；结局在开局或关键节点**预先生成隐藏预测**，再在深层背景节点触发时**动态修改主基调**，使结局既有伏笔又具备动态调整空间，而非固定脚本。

2. **信息差驱动的隐藏剧情**  
   用「未发现信息差条数」达到阈值触发**一次性的隐藏剧情深化**（揭秘、反转），并约束「自然嵌入常规剧情、不直接抛设定」，把「隐藏信息 → 爽点释放」做成可重复、可配置的机制，而非单点彩蛋。

3. **选项与深层背景的显式绑定**  
   在 prompt 中要求「部分选项必须关联角色深层背景」并在输出中带【深层背景关联】标记；选择后**解锁**该角色的深层背景，并强制**后续剧情围绕已解锁深层背景展开**。这样把「选择—解锁—后续叙事约束」做成可解析、可执行的闭环，而不是仅靠 LLM 自由发挥。

4. **配角「展示后建档」+ 身份揭示**  
   配角档案在**展示后触发**建档，避免未出场角色凭空生成；同时用 LLM 从剧情中**抽取身份揭示**（如「黑衣人就是艾玛」）更新配角别名与档案，使配角体系既动态又和剧情强绑定。

5. **预生成两层 + 预生成图片的 Web 体验**  
   在用户看到当前层时后台预生成**下一层 + 再下一层**的选项与剧情（并可带图），选择后直接命中缓存返回，把「等生成」转化为「选完即出」，且与**缓存清理、已用选项清理、线程安全**一起设计，兼顾体验与资源占用。

6. **多基调的细粒度叙事规则**  
   每种基调（HE/BE/NE、黑暗宿命、幽默、唯美、逻辑推理、克苏鲁式等）不仅是一句描述，而是**叙事逻辑、文风、角色规则、关系处理、冲突解决、主题取向、细节要求、禁忌**等多维规则，直接参与世界观与剧情 prompt 的构建，使「选什么基调就有什么风格」可执行、可复现。

7. **现实题材的 Wikipedia 注入**  
   将 Wikipedia 主题与人物检索结果作为**证据块**注入 LLM 与图像提示词，在保持通用剧情生成的前提下，对现实/IP 题材做**定向增强**，属于在通用框架上叠加领域增强的清晰设计。

## 七、全游戏运行逻辑（便于画流程图）

以下按**时间顺序**描述从启动到结局的完整运行逻辑，便于据此绘制游戏运行流程图。

### 7.1 入口与主菜单

1. **启动**：用户执行 `main.py` / `main2.py`（CLI）或 `game_server.py` 后打开前端（Web），进入**主菜单**。
2. **主菜单分支**：
   - **新游戏** → 进入 7.2；
   - **加载存档** → 读档得到 `global_state` 与当前进度，直接进入 7.4 的「当前场景 + 选项」；
   - **存档管理**（列表/删除）→ 操作后回到主菜单；
   - **退出** → 结束。

### 7.2 新游戏：参数输入

3. 用户依次选择或输入：**主角属性**（如性别、年龄、性格等）→ **难度**（简单/中等/困难）→ **故事基调**（如 HE/BE/NE、黑暗、幽默等）→ **主题**（一句话或关键词）；Web 端还可选**图片风格**。
4. 参数齐备后触发**世界观生成**（7.3）。

### 7.3 世界观生成与展示

5. 调用 `llm_generate_global(主题, 主角属性, 难度, 基调)`（可能命中**世界观缓存**或**模板**，未命中则请求 LLM），得到 `global_state`（含 `core_worldview`、`flow_worldline`、`tone` 等）。
6. 可选：**后台异步补全**完整世界观（template 分阶段）。
7. 调用 `generate_ending_prediction()` 生成**隐藏的结局预测**，写入状态供后续基调修改使用。
8. **展示核心设定**给用户（主线、章节矛盾、主角/配角概要等）。
9. **主角立绘**：依赖世界观异步生成主角三视图，写入 `initial/main_character/<game_id>/`。
10. 生成**第一层剧情**：`llm_generate_local` 或等价逻辑得到「开始游戏」的**初始场景 + 2 个选项**。
11. **Web 特有**：此时可请求 **pregenerate-next-layers**，后台预生成**当前 2 选项的下一层 + 再下一层**（含可选场景图），写入 `pregeneration_cache`。
12. 进入**玩法循环**（7.4）。

### 7.4 玩法循环（核心）

**剧情生成依赖（步骤 15「取下一段内容」及 7.3 中第一层剧情均适用）**：每一步剧情生成（`llm_generate_local` / `_generate_single_option`）**强依赖**两类输入——（1）**世界观**：`global_state` 中的 `core_worldview`（核心世界观）、`flow_worldline`（当前章节与矛盾状态）、`tone`（故事基调），要求剧情完全符合世界观与当前状态；（2）**人物形象（主角规范）**：`global_state` 中的 `protagonist_canonical`（主角性别、年龄、外貌等文字设定），用于约束剧情中对主角的描写与主角立绘一致。主角立绘图片本身不参与剧情 LLM 调用，但与剧情共用同一套主角规范，叙事与视觉保持一致。画流程图时可在「世界观生成」「主角立绘」到「第一层剧情」「取下一段内容」之间标出依赖箭头，体现**先有世界观与主角规范，再生成剧情**的顺序。

13. **展示当前状态**：当前场景描述、2 个选项（及可选场景图）、角色/进度面板等。
14. **用户做出选择**：选择选项 A 或 B（或 Web 端「下一段」等）。
15. **取下一段内容**：
    - **Web**：若预生成缓存中有对应 scene_id 的下一层数据，直接**从缓存取**；否则请求 **generate-option** 实时生成。
    - **CLI**：直接调用 `generate_all_options` / `llm_generate_local` 等得到新场景 + 新 2 选项。
16. **更新状态**：用本段输出的 `flow_update` 更新 `global_state['flow_worldline']`（如当前章节、`chapter_conflict_solved`）；若有**深层背景关联**选项被选中，则**解锁**对应角色的深层背景并记录信息差，必要时调用 `modify_ending_tone(触发事件)` 修改结局基调。
17. **配角处理**：从本段剧情中识别**出场/提及配角**，若为首次展示则触发**建档**；可选：生成配角参考图、从剧情中抽取**身份揭示**更新档案。
18. **章节检查**：若本段标记「章节矛盾：已解决」，则 `_check_chapter_conflict()`：当前章节完成 → 进入**章节深化**（对涉及角色做深层背景深化），并可将 `current_chapter` 递增、重置 `chapter_conflict_solved`。
19. **信息差检查**：若**未发现的信息差数量**达到阈值（如 5 条），调用 LLM 生成**隐藏的剧情深化内容**（揭秘/反转），自然插入后续展示。
20. **预生成（Web）**：用户看到新场景与 2 选项后，前端可再次请求 **pregenerate-next-layers**，后台为当前 2 选项预生成下一层 + 再下一层，并执行 **cleanup_old_cache** / **cleanup_used_options** 清理已用缓存。
21. **结局判断**：若满足结局触发条件（如全部章节完成、用户选择结束等），则跳出循环，进入 **7.5**；否则回到步骤 13，用**新场景 + 新 2 选项**继续循环。

### 7.5 结局与返回

22. 调用 `generate_ending_prediction` / `modify_ending_content`（按当前状态）生成**结局文本**。
23. **展示结局**（HE/BE/NE 等主基调 + 具体内容）。
24. 结束后可回到**主菜单**（步骤 2）：新游戏、加载、存档管理或退出。

### 7.6 存档与静态资源（贯穿全程）

- **存档**：玩法循环中或主菜单可**保存**（将 `global_state`、当前场景、选项等写入 `saves/`）；**加载**即读档后回到 7.4 的对应场景。
- **静态资源**：Web 端请求主角图、场景缓存图时，由 **/initial/main_character/...**、**/image_cache/...** 等路由返回文件。

---

**画图建议**：可将「主菜单」画为起点与回流节点；「新游戏 → 参数输入 → 世界观生成 → 展示设定 → 第一层剧情」画为一条竖线；「玩法循环」画为一个**大循环框**，框内依次为：展示 → 用户选择 → 取内容（缓存/实时）→ 更新状态 → 配角/章节/信息差处理 → 预生成（Web）→ 结局判断（是→出框到结局，否→回到展示）；「结局展示」后箭头指回主菜单。存档/加载可画为与主菜单和玩法循环相连的侧枝。
